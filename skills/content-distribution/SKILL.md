# 星期一研究室内容分发渠道

使用这个 Skill，把一篇内容拆成适合不同平台的版本，并确保所有分发最终服务品牌心智、课程转化、B 端线索和知识库沉淀。

## 基本原则

星期一研究室不做全平台平均用力。

每篇内容都应该先判断：

1. 它是建立信任、获取流量、承接搜索、证明能力，还是转化线索？
2. 它更适合长文、卡片、问答、thread、Demo、开源仓库，还是案例复盘？
3. 它最终应该沉淀到 AI 产品测评知识库、信息美学家专栏、课程案例库、GitHub 仓库，还是 B 端方案资产？

内容分发不是简单复制，而是：

> 同一个观点内核，按平台语境改写成不同入口，最终回到统一资产池。

## 平台角色

### 公众号

定位：信任与转化主阵地。

适合发布：

- 完整长文
- 方法论文章
- 工作流案例
- 白皮书连载
- 信息美学家专栏
- 课程拆解
- B 端培训、诊断和管理中枢案例

写法重点：

- 讲清楚问题、过程、判断和最终产物。
- 证明星期一研究室不是工具教程号，而是 AI 工作流与可视化产物设计品牌。
- 文末使用柔性 CTA，导向 AI 产品测评知识库、信息美学家专栏、玩转飞书系统课程或企业服务。

### 小红书

定位：破圈、痛点钩子和视觉化获客。

适合发布：

- 前后对比卡片
- 流程图
- 截图拆解
- 课程片段
- 模板局部展示
- 信息美学视觉案例
- AI 工作流改造前后对比

写法重点：

- 用具体痛点开头。
- 少讲完整理论，多展示结果和变化。
- 每篇只承接一个问题，例如“AI 输出之后放哪里”“飞书怎么变成管理中枢”“项目看板为什么没人用”。

### 知乎

定位：搜索长尾和专业背书。

适合发布：

- 问题式长文
- 行业 AI 落地回答
- 飞书、知识库、项目管理、Agent、AI 工具选型类内容
- 公众号长文的搜索友好改写版

写法重点：

- 标题要像用户会搜索的问题。
- 开头先给判断，再展开方法。
- 结尾导向知识库、课程或企业诊断，不要写得像硬广。

### 少数派

定位：高质量工具用户与系统搭建心智。

适合发布：

- 深度工具使用案例
- 个人工作系统
- AI + 飞书 / Notion / WPS 工作流
- 知识库、项目管理、内容生产系统
- 信息美学和最终产物展示

写法重点：

- 强调真实使用过程、系统结构和复盘。
- 少写泛趋势，多写“我怎么搭、为什么这样搭、最终解决了什么”。

### X

定位：AI Native 洞察、国际工具圈坐标和 build in public。

适合发布：

- AI Native 判断
- Agent / workflow / product thinking
- Demo 展示
- Skill 或工作流发布
- 英文或中英混合 thread

写法重点：

- X 发判断，不发完整解释。
- 信息密度高，结论更锋利。
- 适合把公众号长文压缩成 5 到 8 条 thread。
- 可用英文表达，让观点进入更大的 AI 工具语境。

### 即刻

定位：中文创作者、工具用户和同行交流的过程场。

适合发布：

- 正在做什么
- 今天的工具实验
- 工作流搭建过程
- 文章背后的思考
- 轻量复盘
- 和中文语境强相关的判断

写法重点：

- 即刻发过程，不照搬 X。
- 可以更口语、更现场、更个人。
- 同一观点在 X 上像方法论判断，在即刻上像工作台旁边的思考记录。

### GitHub

定位：能力证明和开源资产沉淀。

适合发布：

- 开源 Skill
- 工作流包
- Demo 项目
- 示例数据
- README 型案例说明

写法重点：

- 每个项目都要说明适合谁、解决什么问题、输入是什么、输出是什么、如何运行、企业版可以怎么定制。
- GitHub 不是主要流量平台，而是证明“我真的能交付系统”的证据。

## 平台优先级

默认优先级：

1. 公众号：完整长文首发，建立信任和转化。
2. 小红书：拆卡片和痛点笔记，获取更广泛的工作场景用户。
3. 知乎：改成问题式长尾内容，承接搜索。
4. 少数派：挑选系统性强的精品内容同步。
5. GitHub：有 Skill、Demo、工作流包时同步沉淀。
6. X / 即刻：持续发布观点、过程和 build in public。

暂缓重投入：

- 抖音
- 视频号
- B 站

这些平台可以后置，等长文、案例和视觉资产稳定后，再改成短视频脚本或直播主题。

## 一稿多投流程

一篇完整内容推荐拆成：

1. 公众号：完整文章。
2. 小红书：3 到 5 篇卡片笔记，每篇只讲一个痛点或一个变化。
3. 知乎：1 篇问题式长尾文章或 2 到 3 个问答。
4. 少数派：只同步系统性强、工具过程清楚的版本。
5. X：1 条主观点 + 1 条 thread。
6. 即刻：1 到 3 条过程记录或个人观察。
7. GitHub：如果文章有 Skill、模板、Demo 或工作流包，同步 README 和示例。

## X 和即刻的差异

不要把 X 和即刻写成完全一样。

- X：发判断、框架、Demo、英文或中英混合观点。
- 即刻：发过程、现场感、中文语境、创作者工作台观察。

同一个观点可以这样改写：

X：

> AI tools are not the final output. The real value is turning AI output into reusable workflows, knowledge systems, and visual assets.

即刻：

> 最近越来越觉得，AI 产品测评不能只写“好不好用”。真正重要的是：它生成的东西最后进了哪里？是停在聊天窗口，还是进了知识库、项目管理、客户交付、课程材料？

## 输出要求

当用户提供一篇文章或选题时，输出：

1. 推荐发布平台和优先级。
2. 每个平台承担的商业角色。
3. 每个平台的标题 / 开头 / 内容形态建议。
4. 是否需要拆卡片、thread、问答、README 或 Demo。
5. 各平台 CTA 应该导向哪里。
6. 最终沉淀到哪个资产池。

默认资产池包括：

- AI 产品测评知识库
- 信息美学家专栏
- 玩转飞书系统课程案例库
- GitHub 开源 Skill / 工作流包
- 企业 AI 工作流诊断案例
- 企业管理中枢方案资产
