# 星期一研究室信息美学家专栏

使用这个 Skill，把「信息美学家」专栏写成一个个轻量但完整的信息整理案例。

这个 Skill 负责栏目定位、结构、标题层级和表达边界。通用语气可配合 `mondaylab-professional-article-style`，品牌方向可配合 `mondaylab-content-direction`。

## 栏目定位

「信息美学家」不是 AI 产品测评，也不是单纯的飞书教程。

它的核心是：把零散信息、素材、任务、知识、灵感或项目过程，整理成一个更清楚、更顺手、更有审美秩序的东西。

可视化产物可以是：

- 飞书多维表格、画册视图、看板视图、知识库页面。
- YouMind 知识卡片、秒哒应用、HTML 页面、落地页。
- Markdown 文档、README、教程文档、说明书、知识库目录。
- 图片素材库、项目案例库、课程素材库、Prompt 库、工作流库。
- 个人品牌资产、团队协作资产、企业知识资产。

工具只是承载容器，不要让飞书、AI 或某个产品喧宾夺主。文章要让读者看到：原本散乱的信息，如何一步步变得更清楚、更好看、更容易被再次使用。

## 第 012 期沉淀出的写法

第 012 期是一个“AI 视觉素材库”案例，不要把它的素材图、字段表、多维表格结构误当成所有选题的固定模板。它真正值得复用的是栏目口吻、段落推进和轻量场景写法：

- 从“大而全的 AI 工作流”收回到一个小场景：AI 生成图片越来越多，如何整理成视觉素材库。
- 从“讲方法论”改成“解决一个具体麻烦”：素材越存越乱，下次找不到。
- 从“强调 AI Native”改成“轻轻带出 AI”：AI 是素材变多的背景，不是全文主角。
- 从“平台能力展示”改成“最终产物展示”：重点是轻量素材库和画册视图，而不是飞书功能清单。
- 从“抽象结论”改成“具体动作”：建表、补字段、上传图片、切画册视图。
- 从“商业化 CTA”改成“柔和收口”：课程只放在文末，不要中途打断案例。
- 从“专业大词”改成“周刊语气”：更像小编给小伙伴分享一个可直接照做的小系统。

以后遇到不同选题时，优先学习它“怎么说话”和“怎么把段落接起来”，而不是照搬它的字段和截图要求。

## 标题策略

标题要同时包含痛点和可视化产物。

优先使用这种结构：

- `第{期数}期 ▷ {真实问题}，怎么整理成{好看/好找/好用/可复用}的{产物}?`
- 副标题可补充工具和场景：`AI 生成的图片越存越乱？用多维表格做一个审美在线的素材画册`

标题不要只写工具名，例如不要写：

- `飞书画册视图教程`
- `AI 视觉资产管理方法论`
- `如何用多维表格管理图片`

要让读者第一眼知道自己遇到的问题，以及这期会产出什么。

## 默认结构

信息美学家专栏默认控制在 1200-2000 字，适合公众号周刊、小红书拆条和知识库沉淀。

推荐结构可以根据选题调整，不必每期都完全一样：

1. 开头：说明这是《信息美学家》第几期，用一句话交代本期做什么小场景。
2. 问题：用读者熟悉的具体麻烦开场，例如图片太多、任务太散、灵感找不到、项目截图难复用。
3. 产物：说明这期要搭出的轻量系统，例如视觉素材库、项目案例库、课程素材库。
4. 展开：按 2-4 个小段落展示怎么整理、怎么观察、怎么搭建或怎么判断。
5. 适用场景：说明哪些人、哪些团队、哪些内容场景可以复用；必要时再用表格。
6. 结尾：回到“可找、可看、可复用”，再轻量引导课程或案例库。

标题层级直接采用第 012 期这种轻量结构，不需要再配合 `article-heading-structure`。默认用法：

- `## （1）{问题或场景}`
- `## （2）{这次做什么}`
- `## （3）{具体怎么做}`
- `#### 第一步：{动作}`
- `#### 第二步：{动作}`
- `## （4）{适合谁}`
- `## （5）结语：{回到价值}`

不要使用专业测评文章里的 `# 一、{emoji}{标题}`、`## 1、{标题}` 那套层级。信息美学家更像轻量周刊栏目，标题要短、松弛、顺着读者的问题往下走。

## 内容写法

先从生活化或工作流里的具体问题开始，不要直接上理论。

可以使用这些进入方式：

- “经常有小伙伴问小编……”
- “如果手上已经有很多……怎么整理成……？”
- “以前我们面临的问题是缺素材，而现在的问题是……”

写正文时，不要把段落写成彼此独立的说明书。段落之间要有自然递进：

- 先写“为什么会遇到这个问题”。
- 再写“这次我们用一个小方法处理它”。
- 接着写“具体处理时，先做哪一步，再做哪一步”。
- 最后写“做完之后，信息的状态发生了什么变化”。

常用衔接方式：

- 从问题到方案：`所以这一期，我们先不做复杂系统，而是从一个很小的整理动作开始。`
- 从场景到步骤：`这个场景拆开来看，其实只需要先处理三件事。`
- 从步骤到效果：`这一步完成之后，原本散在各处的信息，就开始有了一个可以继续补充的位置。`
- 从工具到价值：`工具本身不是重点，重点是它让这些内容之后还能被找回来、看得懂、用得上。`
- 从案例到复用：`如果换成别的场景，这套思路也可以继续迁移。`

如果是操作型文章，每一步可以回答：

- 这一步在搭什么？
- 这一步让信息多了什么结构？
- 读者照做时需要填哪些字段、上传什么素材、切换什么视图？
- 做完后能看到什么结果？

如果是观察型、审美型或案例型文章，不必强行写步骤。可以改成回答：

- 这个信息为什么原本不好读、不好找或不好复用？
- 它被重新整理后，视觉秩序发生了什么变化？
- 哪个细节让它变得更清楚、更舒服或更容易使用？
- 读者可以把这个观察迁移到哪里？

如果是 AI 相关场景，AI 只承担一个具体角色：

- 生成素材。
- 提取字段。
- 生成 Prompt 摘要。
- 帮忙分类、命名、打标签。
- 把结果沉淀进可视化容器。

不要把每篇都写成“AI 时代的人机协同方法论”。这个方向可以存在，但要藏在具体案例后面。

## 选题适配

信息美学家未来可以有很多类型，不要被第 012 期的多维表格素材库限制住。

常见选题类型：

- 工具搭建型：用飞书、多维表格、YouMind、秒哒、HTML 等搭一个轻量系统。
- 审美观察型：拆解某个信息页面、知识库、卡片、表格、看板为什么好看或不好用。
- HTML 美学型：拆解或改造一个 HTML 页面、落地页、作品集、工具页、信息页的排版、留白、层级、组件和可读性。
- Markdown 美学型：优化 Markdown 文档、README、教程、知识库文章的标题层级、列表、表格、引用、代码块和阅读节奏。
- 文档排版型：处理长文、说明书、课程讲义、项目文档、SOP 的结构、重点标注、段落呼吸感和视觉秩序。
- 素材整理型：整理图片、截图、Prompt、案例、课程素材、灵感参考。
- 工作流沉淀型：把一个内容生产、项目管理、知识管理流程整理成可复用结构。
- 案例复盘型：展示一个已经做完的页面、文档、表格或知识库，讲它怎么从混乱变清楚。
- 迁移改造型：把一个场景从飞书迁移到 YouMind、秒哒、HTML、WPS、钉钉、企微等其他容器。

不同选题的“证据”不同，不要统一要求字段表或画册视图。根据场景选择最合适的支撑材料：

- 工具搭建型：可以给字段、视图、页面结构、模板链接。
- 审美观察型：可以给前后对比、局部细节、设计判断、改造建议。
- HTML 美学型：可以给页面截图、区块拆解、结构说明、样式建议、可复用 HTML/CSS 片段。
- Markdown 美学型：可以给原文片段、优化后结构、标题层级、表格样式、列表节奏、可复用 Markdown 模板。
- 文档排版型：可以给目录重组、重点标注、段落拆分、信息分组、阅读路径建议。
- 工作流沉淀型：可以给流程图、步骤清单、节点说明、最终产物截图。
- 案例复盘型：可以给原始状态、整理过程、最终效果、可复用经验。
- 纯观点型：也可以没有完整工具产物，但要有具体例子、具体画面或具体判断。

## 证据和产物

信息美学家最好让读者看见某种“整理后的结果”，但这个结果不一定每次都是多维表格、画册或截图。

可以是：

- 一个页面。
- 一张图。
- 一个表格。
- 一个知识库目录。
- 一套字段。
- 一个 Prompt。
- 一个工作流。
- 一段 HTML/CSS。
- 一个 Markdown 模板。
- 一个文档结构。
- 一个前后对比。
- 一段具体改造建议。

如果这一期是素材库、项目库、课程库这类结构化场景，再优先提供：

- 素材名称
- 素材标签
- 图片附件
- 使用场景
- 生成工具
- Prompt 摘要
- 发布状态
- 适用渠道
- 可复用建议

图片、截图或示意图不要只是装饰。只要出现视觉材料，就要帮助读者理解：原来的信息是什么样，整理后变成了什么样，为什么这样更好用。

如果写作时判断某个位置需要用户补截图，但当前没有图片素材，直接保留截图占位，不要编造图片内容。统一格式：

```text
【📷截图：截图说明】
```

截图说明要写清楚需要截什么、为什么放在这里。例如：

- `【📷截图：多维表格的字段设置页面，重点展示素材名称、标签和图片附件三个字段】`
- `【📷截图：画册视图最终效果，展示图片卡片如何按素材名称排列】`
- `【📷截图：HTML 页面首屏效果，重点展示标题层级、留白和信息卡片布局】`
- `【📷截图：Markdown 优化前后的目录对比，展示标题层级变清楚后的阅读路径】`

截图占位要放在最接近说明文字的位置，不要集中堆到文末。

## 语气边界

可以保留轻松周刊感：

- “小伙伴”
- “小编”
- “咯”
- “跑起来啦”
- “可以直接拿去用”

但要克制使用，不要变得卖萌或口水化。

段落口吻要像是在陪读者一起整理一个小东西：

- 先承认问题很常见，不要上来批判用户不会管理。
- 先给一个小切口，不要一开始就说“系统化”“底层逻辑”。
- 多用“这一步做完之后”“接着”“如果你平时也会……”这类自然连接。
- 结论要落在具体体验上，例如“下次就不用靠记忆翻文件夹了”，不要落在空泛价值上。

避免这些写法：

- 把信息美学家写成工具说明书。
- 把一个轻量场景拔高成宏大判断。
- 过早出现课程销售、企业培训、商业化方案。
- 连续使用“底层逻辑”“人机协同”“系统化”“最终产物”等抽象词。
- 只写工具功能，不展示具体例子、具体画面或具体判断。
- 用“不是……而是……”“不在于……而在于……”这类 AI 腔做结论。

## 结尾和 CTA

结尾先收束本期案例，不要先卖课。

推荐顺序：

1. 总结这个小系统解决了什么问题。
2. 说明素材、任务或知识进入结构后，如何变得更好找、更好看、更好复用。
3. 如果需要 CTA，再轻量提到《玩转飞书系统课程》或“信息美学家案例库”。

CTA 示例：

```text
如果你想系统学习如何用飞书搭建知识库、项目管理和内容系统，可以继续文末的《玩转飞书系统课程》。

如果只是想先积累灵感，也可以把这一期放进你的“信息美学家”案例库里。
```

## 输出标准

- 保持中文 Markdown。
- 优先写成一篇能直接发布的专栏文章，而不是写作建议。
- 如果用户提供原稿，先判断它是否像「信息美学家」；不像时，指出偏离点再改。
- 如果用户只给场景，直接补出标题、结构、字段、截图建议和正文初稿。
- 如果文章涉及飞书多维表格、知识库或画册视图，优先给出可落地字段和视图设计。
- 最终文章要让读者觉得“我也能照着搭一个”，而不是只觉得“这个观点很对”。
